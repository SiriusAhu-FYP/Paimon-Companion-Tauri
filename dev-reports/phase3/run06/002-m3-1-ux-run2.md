# M3.1 UX 收尾 Run 2 报告

## 本次目标

修复用户反馈的两个 UX 问题：
1. 旧 LLM/TTS 手动配置区冗余，与 Profiles 区并存造成混乱
2. L2D 默认置顶导致 Settings 面板被遮挡无法操作

## 本次完成内容

### 1. 旧 LLM/TTS 手动配置区替换为最小测试区

**变更：**
- 移除旧的"LLM 设置"手动配置区（Provider、Base URL、API Key、Temperature 等字段）
- 移除旧的"TTS 设置"手动配置区（Provider、GPT 权重路径、服务地址等字段）
- 两个区域均替换为极简测试区，只显示：
  - 当前读取来源（激活档案名 或 "根配置"）
  - 测试连接按钮（从激活 Profile 读取配置）
  - TTS 直测输入框和合成按钮
- 测试逻辑从 `getActiveLlmConfig()` / `getActiveTtsConfig()` 读取，确保优先读激活档案，fallback 到根配置

**交互模型：**
- 所有详细配置（Base URL、模型、权重路径等）统一在 Profile Dialog 中管理
- Settings 面板只保留测试能力，不再做配置编辑

### 2. L2D 置顶遮挡 Settings 问题

**变更：**
- `SettingsPanel` 打开时，useEffect 发送 `broadcastControl({ type: "set-always-on-top", value: false })` 强制解除置顶
- 关闭时（useEffect cleanup）发送 `broadcastControl({ type: "restore-always-on-top" })` 恢复之前的置顶状态
- `StageWindow` 新增 `restore-always-on-top` case：仅在 floating 模式下根据 `alwaysOnTop` 状态恢复
- `ControlCommand` 类型新增 `restore-always-on-top` 分支
- `handleControlCommand` 依赖数组加入 `stageMode` 和 `alwaysOnTop` 以确保正确恢复

- Profile 选择后尚未联动到实际 LLM/TTS provider 实例（仅存储配置）

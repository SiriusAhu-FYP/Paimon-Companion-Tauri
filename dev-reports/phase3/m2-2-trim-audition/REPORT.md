## 3. 实验设计

### 3.1 基础三组实验

#### 方案 A：三句话都不裁切
- 保留原始音频
- 直接按顺序拼接
- 输出：`concat_raw.wav`

#### 方案 B：三句话都裁切首尾
- 对每句都做前导静音裁切 + 尾部静音裁切
- 保留少量安全 margin，避免切得过硬
- 输出：`concat_trim_both.wav`

#### 方案 C：三句话都只裁切尾部
- 不裁前导
- 只裁每句结尾静音
- 尾部同样保留少量安全 margin
- 输出：`concat_trim_tail_only.wav`

### 3.2 补充实验

#### 方案 D：前导轻裁 + 尾部重裁
- 前导只轻微裁切（保留一点空气感）
- 尾部较明显裁切
- 输出：`concat_trim_front_light_tail_heavy.wav`

## 4. 裁切设置

| 设置项 | 值 | 说明 |
|--------|-----|------|
| 静音阈值 | 0.01 | 样本绝对值 / 32767，用于判断静音 |
| 前导保护边缘（方案 B） | 10ms | 保留少量前导静音，避免切得过硬 |
| 尾部保护边缘（方案 B、C） | 20ms | 保留少量尾部静音，避免切得过硬 |
| 前导轻裁保护边缘（方案 D） | 20ms | 保留更多前导静音，增加空气感 |
| 尾部重裁保护边缘（方案 D） | 10ms | 减少尾部静音，使拼接更紧凑 |

**设置理由**：
- 静音阈值 0.01 是一个平衡值，既能检测到明显的静音，又不会误裁低声量语音
- 前导保护边缘 10-20ms 可以保留自然的呼吸感，避免语音突兀开始
- 尾部保护边缘 10-20ms 可以保留句尾的自然停顿，避免语音突兀结束

## 7. 听感预判

### 7.1 风险点分析

1. **原始音频（方案 A）**：
   - 风险：尾部静音过长，导致句子间停顿明显，可能产生卡顿感
   - 优势：保留完整的自然停顿，语音听起来更自然

2. **裁切首尾（方案 B）**：
   - 风险：可能会切掉部分自然的呼吸和停顿，使语音听起来有些急促
   - 优势：句子间过渡更紧凑，减少卡顿感

3. **只裁尾部（方案 C）**：
   - 风险：前导静音保留完整，可能导致句子开始前有短暂停顿
   - 优势：既保留了语音开始的自然感，又减少了句子间的停顿

4. **前导轻裁+尾部重裁（方案 D）**：
   - 风险：尾部裁切较狠，可能导致句尾停顿不足
   - 优势：前导保留适当空气感，尾部裁切使拼接更紧凑

### 7.2 卡顿主因分析

根据实验数据，当前卡顿感更接近：
- **尾部静音过长**：尾部静音平均 358ms，明显长于前导静音（平均 54ms）
- 合成耗时：700-1500ms，确实是一个瓶颈，但通过预缓冲可以缓解

## 9. 实验结论

### 9.1 裁切策略推荐

**推荐方案**：暂不推荐将 trim 作为默认正式策略

**推荐理由**：
- 虽然实验数据显示裁切可以有效减少 10% 左右的总时长
- 但人工听感判断表明，完全不裁切的原始版本反而更自然
- 尾部静音虽然较长，但保留了自然的停顿感

### 9.2 是否建议进入正式实现

**建议**：暂不进入正式实现

**理由**：
- 人工听感优先于数据指标
- 虽然 trim 缩短了时长，但在当前拼接听感中，raw 版本更自然
- 现阶段更怀疑问题主因是"合成等待 / 预缓冲 / 队列调度"，而不是单纯尾部静音
- trim 实验结果保留，但暂不作为默认正式方案

### 9.3 人工听感修正说明

**修正说明**：
- 当前人工听感优先
- 虽然 trim 缩短了时长，但在当前拼接听感中，raw 版本更自然
- 因此现阶段更怀疑问题主因是"合成等待 / 预缓冲 / 队列调度"，而不是单纯尾部静音
- trim 实验结果保留，但暂不作为默认正式方案

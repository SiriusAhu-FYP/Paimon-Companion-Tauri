# Paimon Live 中 MCP 的定位与边界

## 1. 本文档的目的

本文档用于明确：

- MCP 在 Paimon Live 中到底承担什么角色
- 哪些能力适合放进 MCP
- 哪些能力不适合放进 MCP
- MCP 与本地执行内核、本地服务、未来 Skill 层之间的关系

本文档的目标不是介绍 MCP 通识，而是结合 **Paimon Live 当前的产品形态**，给出可执行的边界定义，避免后续开发中把“适合做成 MCP 的能力”和“必须保留在本地执行内核中的能力”混在一起。

---

## 2. 当前项目语境

Paimon Live 不是一个单纯的聊天前端，也不是一个只做知识问答的 agent 桌面壳。

它当前已经形成以下基本结构：

- 主窗口负责控制、监测、配置与人工干预
- Stage 是独立播出窗口，也是 OBS 的正式捕获对象
- Stage-only 单实例播出架构已经成立
- 本地应用内部已经有运行时控制、Pipeline、TTS 播放、口型、Stage 窗口与角色驱动等执行链路
- 后续将逐步接入真实 LLM、真实 TTS、角色卡、知识层、直播间输入、商品资料与外部能力

在这种语境下，MCP 不能再被简单理解为“额外接一个知识库接口”。

对于 Paimon Live，MCP 更适合被理解为：

**一个为主播行为提供标准化能力接口的层。**

---

## 3. Paimon Live 中的 MCP 是什么

在本项目中，MCP 更适合承担以下角色：

### 3.1 标准化能力暴露层
用于把某些能力以一致方式暴露给模型或编排层，例如：

- 商品信息查询
- 角色设定读取
- 当前直播上下文读取
- FAQ / 运营话术检索
- 离散动作请求
- 外部系统查询与操作

### 3.2 外部知识与外部动作的接口层
MCP 更适合提供：

- Resources：上下文数据、资料、设定、知识结果
- Tools：离散动作或可调用函数
- Prompts：模板化的结构化提示入口

### 3.3 主播决策的“能力池”
MCP 不一定直接执行所有事情，但它可以为模型/Skill/编排层提供“我能查什么、我能做什么”的能力目录。

---

## 4. MCP 不是什么

在 Paimon Live 中，MCP **不是**：

- Live2D 渲染引擎
- Stage 窗口控制层
- 音频播放内核
- 口型逐帧驱动系统
- 播放队列与合成队列本体
- 运行时状态机本体
- 急停与人工接管底层执行器

也就是说：

**MCP 不负责高频执行，不负责底层实时控制，不负责本地播出内核。**

---

## 5. 适合放进 MCP 的能力

## 5.1 知识与上下文能力

这是最适合先放进 MCP 的部分。

例如：

- 当前商品资料
- 商品 FAQ
- 角色设定
- 当前直播上下文
- 活动话术
- 当前场次特殊要求
- 后续 RAG 检索结果

适合做成：

- Resource：当前商品、角色 profile、live context
- Tool：按条件检索 FAQ、按 query 检索知识
- Prompt：基于商品/角色生成某类标准话术

### 原因
这些能力本质上是在“给模型补上下文”，而不是在高频实时控制舞台。

---

## 5.2 离散业务动作

例如：

- 切换当前商品
- 发送一条弹幕回复
- 查询库存或价格
- 查询直播状态
- 请求切换某个表情
- 请求触发某个动作

这里的关键词是：

**离散、低频、可容错、可回退。**

### 原因
即使这些动作延迟几百毫秒到几秒，直播场景通常也仍然可接受。  
你们已有经验也说明：只要整体表现自然、稳定、可控，很多行为并不需要极端实时。

---

## 5.3 主播工作流中的“决策型动作”

例如：

- 当前这句回复应该用什么情绪
- 是否应该切一个动作
- 当前回复是否应该先查商品知识
- 当前回复是否需要重写成更适合 TTS 的 spoken_text
- 当前是否应该切换商品或插入直播上下文

这类能力很适合由模型通过 MCP Tool 发起调用，或由后续 Skill 层将其组合使用。

### 原因
这些动作更像“决策”而不是“执行”。  
决策可以允许更高延迟，但执行必须稳定。

---

## 6. 不适合放进 MCP 的能力

## 6.1 Stage 实时执行链路

以下能力不适合做成 MCP：

- Stage 窗口位置/attach/docked/floating 底层控制
- Stage 显示层高频同步
- Live2D 渲染器本体
- 角色参数逐帧更新

### 原因
这些属于本地播出内核，要求更高的确定性与一致性。

---

## 6.2 音频播放与口型内核

不适合做成 MCP：

- 音频播放队列
- 合成结果播放调度
- 口型高频参数更新
- speaking 状态的逐段/逐帧变化

### 原因
这些属于执行层。  
它们应由本地 Pipeline、AudioPlayer、CharacterService 等内部服务控制。

---

## 6.3 运行时控制核心

不适合做成 MCP：

- 急停
- 人工接管底层逻辑
- runtime mode 状态机
- 锁麦 / 解锁执行器
- provider 初始化与降级内核

### 原因
这些是应用控制平面的核心能力，不应轻易交给模型自由调用。

---

## 7. 重新定义：哪些“动作”可以走 MCP，哪些不行

为了避免误解，这里明确区分：

### 可以走 MCP 的
- “切 happy 表情”
- “触发一个 wave 动作”
- “切换当前商品”
- “查当前商品 FAQ”
- “把回复改写成更适合口播的 spoken_text”
- “决定是否发送某种运营回复”

### 不应该走 MCP 的
- 每帧更新 mouth 参数
- 每帧驱动表情插值
- 控制 Stage 窗口每次 resize 的底层语义
- 音频 buffer 的实时播放调度

### 边界原则
**MCP 更适合承担“请求和决策”，不适合承担“高频执行”。**

---

## 8. 在 Paimon Live 中推荐的分层方式

建议以后按以下 4 层理解：

## 8.1 执行内核层（本地应用核心）
包含：

- Stage
- CharacterService
- AudioPlayer
- Pipeline
- Runtime
- TTS/播放队列
- 口型驱动
- 窗口语义控制

这一层不应 MCP 化。

---

## 8.2 MCP 能力层
包含：

- 商品信息查询能力
- FAQ / 知识检索能力
- 当前上下文资源
- 角色设定资源
- 离散动作请求能力
- 直播运营工具能力

这一层负责把“能力”标准化暴露出来。

---

## 8.3 Skill / 业务编排层（后续）
包含：

- reply_to_danmaku
- sell_product
- introduce_product
- rewrite_for_tts
- choose_emotion_for_reply
- react_to_gift
- 其他主播业务动作

这一层会组合：
- Prompt
- MCP Resources / Tools
- 本地服务
- fallback 规则
- 状态约束

---

## 8.4 UI / 控制层
包含：

- 主控界面
- 设置页
- 状态监控
- 人工干预与急停入口

---

## 9. 当前最值得优先 MCP 化的部分

我建议 Paimon Live 后续优先考虑以下几类 MCP 能力：

### 9.1 商品/知识 MCP
最优先，最自然。

建议至少包括：
- get_current_product
- get_product_faq
- get_live_context
- get_character_profile
- retrieve_knowledge（后续）

---

### 9.2 直播运营 MCP
第二优先。

建议包括：
- switch_product
- get_inventory
- get_live_room_status
- reply_to_danmaku（后续可审慎纳入）

---

### 9.3 主播行为决策 MCP
可以逐步引入。

建议包括：
- select_emotion_for_reply
- select_motion_for_reply
- rewrite_for_tts
- request_stage_expression
- request_stage_motion

注意：
这里更推荐把它们视为“请求型动作”，由本地执行层最终决定如何落地。

---

## 10. 当前不建议优先 MCP 化的部分

以下能力当前不建议优先做成 MCP：

- 嘴型逐帧控制
- 口型队列
- 音频播放控制
- Stage attach / floating / pinToApp 底层行为
- runtime 底层状态机
- 急停与人工接管底层执行

原因很简单：
这些越往底层，越不适合交给模型按工具调用驱动。

---

## 11. 与后续 Skill 文档的关系

本文档只回答：

- MCP 是什么
- MCP 的边界在哪里
- 哪些能力适合被 MCP 化

本文档**不负责**定义：

- 什么叫 Skill
- Skill 如何组织多个 MCP 能力
- Skill 如何与本地服务、Prompt、fallback 一起工作

这些将在后续 `Skill` 文档中定义。

---

## 12. 结论

在 Paimon Live 中，MCP 的最佳定位不是“知识插件集合”，也不是“实时控制层”。

更准确的说法是：

**MCP 是主播能力的标准化接口层。**

它适合承接：

- 外部知识
- 商品与上下文能力
- 离散业务动作
- 主播行为决策所需的可调用工具

但不适合承接：

- 高实时、高频执行链路
- 本地播出内核
- 音频播放与口型底层控制
- runtime 控制核心

一句话总结：

**MCP 负责“主播可以知道什么、请求什么、决定什么”；本地应用核心负责“主播最终如何稳定地做出来”。**

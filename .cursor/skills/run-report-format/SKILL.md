# Run 报告格式规范

## 核心原则

1. **报告是交付物的一部分**：没有报告的 run，视为交付不完整
2. **先写事实，再写结论**：做了什么、改了什么、验证了什么，而非先写"圆满完成"
3. **必须记录 commit 信息**：commit hash、分支名、报告路径缺一不可
4. **区分已完成 / 未完成 / 延后项**：不把"计划中的"写成"实际完成的"
5. **验证状态必须明确**：代码存在 ≠ 验证通过
6. **风险和限制不能省略**：即使本轮总体成功，也应写明限制和未覆盖范围
7. **格式统一，便于回看**：结构化、短小但完整，便于后续查阅和交接

## 推荐输出结构

```
# [Run 标题]

## 本次目标
- 本轮原本要解决什么问题

## 本次完成内容
- 已完成项（如有必要，分条列出）

## 关键改动
- 关键文件 / 模块
- 每个文件大概改了什么

## 验证情况

| 层次 | 状态 | 说明 | 证据 |
|------|------|------|------|
| 编译 / lint | ✅/❌ |  |  |
| 浏览器端验证 | ✅/❌ |  |  |
| Tauri 桌面端验证 | ✅/❌ |  |  |
| OBS / Stage / Provider 验证 | ✅/❌ |  |  |
| 尚未验证 | （列出未验证项） | — | — |

> 证据示例：`pnpm tauri dev 手测` / `浏览器 devtools` / `OBS 预览` / `用户亲测反馈` / `未提供`

## 风险 / 限制 / 未完成项
- 当前还存在什么限制
- 哪些点延期到后续处理
- 是否有 blocker（如有，标明优先级）

## 结论
- 本轮是否可视为完成
- 是否建议进入下一阶段
- 如果不建议，下一步应补什么

## 元信息
- Commit: `hash` / 未获取
- Branch: `分支名` / 未获取
- 报告路径: `dev-reports/...`
- 相关文档路径: `blueprints/...` / `docs/...` / 无
```

## 各场景输出重点

### 场景 1：实现 run

重点写：
- 本轮目标
- 改动文件（路径 + 改动摘要）
- 验证情况（含各验证层次）
- commit hash
- 是否可进入下一步

### 场景 2：blocker-fix run

重点写：
- blocker 根因（简短描述）
- 修复内容（改了什么文件 / 函数）
- 该 blocker 是已解除 / 部分缓解 / 仅暂时绕过
- 修复后是否验证通过
- 还剩哪些 residual issue

### 场景 3：文档 / blueprint run

重点写：
- 改了哪些文档（路径）
- 边界 / 范围如何变化
- 是否准备进入实施阶段
- commit hash

## 约束

- 结构化、简洁，不写流水账
- 不用大段空话，不把"计划中的内容"写成"实际完成"
- 不省略验证情况和 commit hash
- 如果缺少某项信息，明确写"未获取"或"未验证"
- 每份报告对应一个 commit，不允许无 commit 的 run 报告

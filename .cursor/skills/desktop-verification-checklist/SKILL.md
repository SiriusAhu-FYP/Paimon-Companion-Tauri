# Paimon Live 桌面端验证 Checklist

## 核心原则

1. **真实手测优先于实现报告**：用户亲自验证到的现象，优先于代码审查或汇报文档的描述
2. **区分验证层次**：代码存在 → 编译通过 → 浏览器端可用 → 真实 Tauri 桌面端可用 → OBS/Stage 真实链路可用
3. **从用户视角验证**：关注功能是否真的可用，而非函数是否被调用
4. **优先给 checklist**：输出可执行的核查清单，不空谈
5. **记录现象**：记录通过/不通过、环境、复现步骤、是否稳定复现

## 验证层次

| 层次 | 说明                   | 验证方式                      |
| ---- | ---------------------- | ----------------------------- |
| L1   | 代码存在               | 代码审查                      |
| L2   | 编译/构建通过          | 构建输出                      |
| L3   | 浏览器端可用           | `pnpm dev`                    |
| L4   | 真实桌面端可用         | `pnpm tauri dev` 或打包后运行 |
| L5   | OBS/Stage 真实链路可用 | 真实场景手测                  |

## 输出格式

### 1. 验证目标
（本次要验证什么能力）

### 2. 环境前提
- 需要的运行环境
- 是否需要 Tauri / OBS / provider / API key

### 3. 验证 Checklist

#### 层级 A：Tauri 启动与基础渲染

| 步骤 | 操作                                    | 预期结果                                | 不通过时观察                           |
| ---- | --------------------------------------- | --------------------------------------- | -------------------------------------- |
| A1   | 运行 `pnpm tauri dev`，观察启动日志     | 无 panic，无大量错误日志，窗口出现      | panic 堆栈、窗口未出现                 |
| A2   | 观察主窗口（控制面板）渲染              | 窗口内显示内容，无白屏，所有控件可见    | WebView 初始化失败、空白窗口、控件缺失 |
| A3   | 观察 Stage 窗口渲染（独立窗口或 Panel） | Stage 角色画面正常，文字层/遮罩正常显示 | 黑屏、渲染错乱、层叠异常               |

#### 层级 B：主窗口与 Stage 语义验证

| 步骤 | 操作                                                            | 预期结果                                        | 不通过时观察                               |
| ---- | --------------------------------------------------------------- | ----------------------------------------------- | ------------------------------------------ |
| B1   | 在主窗口触发控制操作（如切换状态、发送消息）                    | Stage 窗口画面对应变化                          | 操作发出但 Stage 无响应                    |
| B2   | 在主窗口进行常见操作（切配置、发消息、切换 provider、切换状态） | 主窗口操作不会意外破坏 Stage 的状态与播出稳定性 | Stage 被遮挡、状态重置、播出中断、画面异常 |
| B3   | Stage 窗口 resize                                               | Stage 布局响应正确                              | 布局错乱、角色变形                         |

#### 层级 C：OBS 捕获 Stage 验证

| 步骤 | 操作                                 | 预期结果                                     | 不通过时观察                        |
| ---- | ------------------------------------ | -------------------------------------------- | ----------------------------------- |
| C1   | OBS 添加窗口捕获，源选 Stage 窗口    | Stage 画面被正确捕获，画面完整               | 黑屏、捕获到错误内容                |
| C2   | OBS 预览 / 录制 / 推流短测（三选一） | Stage 捕获链路稳定，画面持续正常，无明显异常 | 黑屏、冻结、明显掉帧、内容错位      |
| C3   | Stage 画面与 OBS 捕获内容一致        | 字体、动画、角色状态与 Stage 同步            | OBS 捕获内容与 Stage 实际画面不一致 |

#### 层级 D：Provider 与配置验证

| 步骤 | 操作                                 | 预期结果                                         | 不通过时观察                                |
| ---- | ------------------------------------ | ------------------------------------------------ | ------------------------------------------- |
| D1   | 在配置页填入 provider mock 配置      | 配置保存，mock 模式下核心流程可跑通              | 配置未保存、切换后崩溃                      |
| D2   | 切换到 real provider                 | 行为变化（响应来源/速度/格式等），无 panic       | 切换后无变化、panic、报错                   |
| D3   | 填入无效配置                         | 优雅降级或明确错误提示（非 panic）               | 静默失败、panic                             |
| D5   | 修改配置并保存后关闭应用，再重新打开 | 普通配置与敏感配置都能按预期恢复，连接测试仍可用 | 配置丢失、secret 不可用、重启后回退到旧状态 |

#### 层级 E：回复链路与语音验证

| 步骤 | 操作                                         | 预期结果                                    | 不通过时观察                                   |
| ---- | -------------------------------------------- | ------------------------------------------- | ---------------------------------------------- |
| E1   | 触发一次完整对话（发送消息 → 收到回复）      | 收到流式回复，文字逐段出现在 Stage 或主窗口 | 无响应、回复卡住、乱序                         |
| E2   | TTS 播放与嘴型同步                           | 音频播放正常，角色嘴型与语音基本同步        | 无声音、嘴型不同步、音频明显延迟、播放异常卡顿 |
| E3   | 队列积压测试（快速连续发多条）               | 多条消息积压时按序处理，最终全部完成        | 乱序、丢失、卡死在某条                         |
| E4   | 回复过程中切换 Stage 状态（如切换表情/动作） | 状态切换即时响应，不阻断当前回复流          | 状态切换延迟、阻断回复流                       |

#### 层级 F：退出与进程清理

| 步骤 | 操作                               | 预期结果                         | 不通过时观察               |
| ---- | ---------------------------------- | -------------------------------- | -------------------------- |
| F1   | 正常关闭应用（点击关闭或退出按钮） | 窗口关闭，无残留进程             | 进程残留                   |
| F2   | 强制关闭后重新启动                 | 可正常重启，无端口占用、无 panic | 端口占用、panic on startup |

## 结果记录模板

```
## 验证记录

### 能力：[能力名称]
### 验证人：手动验证
### 环境：Tauri vX.X.X / OBS X.X（如有）/ Provider: [real/mock]

### 真实手测现象记录

| 步骤 | 结果 | 真实现象               | Blocker? |
| ---- | ---- | ---------------------- | -------- |
| A1   | ✅/❌  | （描述实际看到的内容） | 是/否    |
| ...  | ...  | ...                    | ...      |

### 总结
- 通过：X 项
- 不通过：X 项
- Blocker：X 项（如有，需优先修复）
- Polish 问题：X 项（可在后续阶段继续）
- 风险项（未验证但怀疑有问题）：X 项
```

**原则：**
- 优先记录"实际看到什么"，而非"代码是否调用"
- 如果实际现象与报告描述不符，以实际为准，在现象栏注明
- Blocker 必须有明确的复现步骤和证据

## 常见场景 Checklist 速查

### 场景 1：代码改动后 Smoke Test

目标：确认核心链路未被破坏

```
- [ ] Tauri 启动正常，无 panic
- [ ] 主窗口（控制面板）渲染正常
- [ ] Stage 窗口渲染正常
- [ ] OBS 可正常捕获 Stage
- [ ] provider mock 模式下核心流程可跑通
- [ ] 退出无残留进程
```

### 场景 2：Provider 接入验证

目标：确认 provider 真实可用

```
- [ ] 配置页可保存 provider 配置
- [ ] mock → real 切换后行为有明确变化
- [ ] real provider 下流式回复正常
- [ ] provider 异常时降级行为符合预期
- [ ] 配置变更后新建对话使用最新配置
- [ ] 保存配置后重启应用，普通配置与敏感配置均可恢复
```

### 场景 3：OBS / Stage 播出链路验证

目标：确认 Stage 是 OBS 的正确捕获对象

```
- [ ] OBS 窗口捕获选择 Stage
- [ ] OBS 推流时 Stage 画面完整
- [ ] Stage 角色/文字/动画层在推流中一致
- [ ] 推流期间主窗口操作不干扰 Stage 稳定性
```

### 场景 4：Close-out 前完整验证

目标：全面确认可放行

```
- [ ] 上述 A-F 各层全部通过
- [ ] 所有 mock → real 切换正常
- [ ] 保存配置并重启后，配置恢复正常
- [ ] OBS 推流连续 5 分钟无明显异常
- [ ] 无 panic/log error 在正常流程中
```

## 约束

- 不直接改代码，只提供验证指导
- 不假设验证环境，要求确认前提条件
- 不在缺少真实环境时假装验证通过
- 不把 checklist 写成模糊建议列表
- 记录时区分 blocker 与 polish 问题
- 真实手测现象优先于代码审查或实现报告的描述

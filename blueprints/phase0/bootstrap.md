# Phase 0 / Bootstrap

## 1. 本阶段的定位

本阶段不是开始重写旧项目，也不是开始接入真实业务服务。

本阶段的目标只有两个：

1. 创建一个**新的项目骨架**，作为后续重构工作的承载仓库
2. 基于旧项目 `VoiceL2D` 与新需求，完成**审计、规划、分阶段设计**

也就是说，这一阶段要做的是：

**先搭“新桌面项目的空壳 + 规划文档系统”，再为后续 phase1/phase2/... 提供清晰可执行的蓝图。**

---

## 2. 项目背景

当前有一个旧项目：`VoiceL2D`，位于 `E:\FYP-PROJECT\core\VoiceL2D-MVP`

旧项目已经验证了一条虚拟角色交互链路能够工作，包括但不限于：

- 语音输入
- VAD / ASR
- 文本传递给 LLM
- LLM 输出回复文本
- 一路送入 TTS 合成语音
- 一路触发表情 / 动作 / 角色反馈
- 前端角色显示与基础控制能力

但是，当前新项目的目标不是继续在旧仓库上修修补补，也不是简单把旧项目搬进 Tauri。

新项目的方向是：

**从 `VoiceL2D` 中提炼出真正有价值的核心能力，重新设计成一个新的 Tauri 桌面应用。**

这个新应用将更接近一个可操作、可演示、可打包的产品化桌面系统，而不是单纯的技术 demo。

---

## 3. 新项目目标（一句话）

新项目要做的是：

**一个由大模型驱动的虚拟 AI 主播桌面系统。**

它需要同时具备：

- 角色显示与状态反馈
- 语音链路（ASR → LLM → TTS）
- 表情 / 动作触发
- 监控与控制面板
- 外部事件输入接口
- 日志与调试能力
- 紧急停止与人工接管
- 透明背景角色输出，供 OBS / 直播软件捕获
- 尽量降低非技术人员的使用门槛
- 最终支持一键启动与打包分发

---

## 4. 本阶段严格限制

### 4.1 本阶段不要做的事

本阶段**禁止**直接开始以下行为：

- 不要大规模迁移旧代码
- 不要在旧仓库中就地爆改
- 不要接入真实 ASR / TTS / LLM 服务
- 不要一开始就保留旧 Python 架构
- 不要把所有业务逻辑默认塞进 Rust
- 不要急着实现虚拟摄像头
- 不要急着实现真正可用的 RAG 检索
- 不要先做 SaaS、多人协作、云平台相关设计
- 不要提前进入性能优化阶段

### 4.2 本阶段允许做的事

本阶段允许：

- 创建新仓库骨架
- 初始化 Tauri 桌面项目
- 创建基础目录结构
- 写规划文档
- 审计旧项目代码与能力边界
- 明确未来模块划分
- 设计数据流 / 事件流 / 状态结构
- 为未来模块预留位置
- 创建 mock / placeholder 级别的目录和空模块，但不实现真实业务

---

## 5. 技术路线默认选择

除非在审计中发现强理由，否则默认采用以下技术方向：

### 5.1 桌面宿主

- 使用 **Tauri**

Tauri 主要负责：

- 桌面宿主
- 窗口管理
- 本地能力
- 打包分发
- 应用启动与整合

### 5.2 前端

- 使用 **React + TypeScript**

### 5.3 包管理器

- 使用 **pnpm**

### 5.4 前端构建工具

- 使用 **Vite**

### 5.5 Rust 的职责

Rust 默认只承担：

- Tauri 宿主层职责
- IPC / 桌面能力桥接
- 打包与本地集成相关职责

### 5.6 TypeScript 的职责

TypeScript 默认优先承担：

- 核心业务逻辑
- 状态管理
- 事件编排
- 控制面板逻辑
- 多模块协调
- 未来可抽离为独立逻辑包的业务层

### 5.7 Python 的态度

Python **不是必须保留**。

只有在下面情况成立时，才考虑保留部分 Python：

- 某模块强依赖 Python 生态
- 迁移成本过高
- 短期重写收益太低

否则，默认倾向迁移到 TypeScript。

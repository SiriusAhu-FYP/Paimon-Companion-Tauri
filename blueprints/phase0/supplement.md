# Phase 0 Supplement

## 1. 本文档的目的

本文档用于补充 `phase0` 阶段中尚未充分说明、但已经会影响后续架构判断与阶段规划的重要设定。

它的作用不是替代已有文档，而是：

- 补充之前没来得及写清的关键约束
- 修正一些容易误解或表述过满的地方
- 为现有 `architecture.md`、`module-design.md`、`data-flow.md`、`phase-roadmap.md` 提供统一修订依据
- 明确 Phase 0 收口后，Phase 1 应如何开始

阅读优先级上，本文档应与 `bootstrap.md` 同等重要。

---

## 2. 项目目标的再次明确

本项目要做的不是：

- 把 `VoiceL2D` 原样搬进 Tauri
- 继续维护旧项目的历史拆分方式
- 先追求完整自动化和全平台兼容
- 先做成复杂云平台或 SaaS

本项目要做的是：

**将 `VoiceL2D` 中已经验证过的关键能力重新组织，做成一个更适合直播场景、可监控、可人工接管、可打包分发的虚拟 AI 主播桌面系统。**

重点是：

- 保留核心能力
- 重做不合适的旧结构
- 让操作者能够像使用后台系统一样控制直播流程
- 让非技术用户也能完成基本操作

---

## 3. 当前阶段新增明确约束

## 3.1 当前目标平台：仅 Windows

当前阶段的目标平台明确为：

- **Windows**

这意味着：

- 现阶段不以 Linux / macOS 兼容为目标
- 文档、实现、测试、路径处理、窗口行为、音频设备行为都可以优先针对 Windows
- 不需要为了“未来可能跨平台”而在当前阶段引入额外复杂度

如果未来要扩展平台，再单独规划，不在当前阶段预支复杂度。

---

## 3.2 “一个桌面程序”不等于“永远只有一个进程”

之前若有“单进程架构”的表述，应修正理解为：

**单桌面宿主应用 + 可插拔外部服务 / 外部依赖**

这里真正要保证的是：

- 用户侧尽量只需要启动一个桌面程序
- 操作者不需要自己手动开多个终端和服务
- 桌面应用统一承担整合、监控、控制、配置与启动职责

而不是强行承诺：

- 所有模块都必须永远运行在同一个系统进程里

因此后续架构上允许存在：

- 外部 ASR 服务
- 外部 TTS 服务
- 外部 LLM API / 本地推理服务
- 未来必要的 sidecar / helper process

但这些应被桌面宿主统一管理和屏蔽复杂度，而不是暴露给非技术操作者。

---

## 3.3 角色系统要追求“单一状态真源”，不是强行“单一渲染实例”

本项目中需要区分两个概念：

### 不应被误解为：
- 一个渲染器硬喂两个窗口
- 两个窗口共享同一个具体 renderer 对象

### 真正应坚持的是：
- **角色状态只有一个权威真源**
- 表情、动作、嘴型、角色当前状态、任务状态等应该来自同一个 authoritative state
- 主控界面预览与直播输出窗口应基于同一状态真源进行同步渲染

换句话说，更准确的目标是：

**单一状态真源，双窗口同步渲染。**

这样做的目的：

- 避免监控窗口和直播窗口状态不一致
- 避免动作不同步
- 避免一个窗口改表情、另一个窗口没变
- 让调试、录制、推流看到的角色状态可对齐

---

## 3.4 “语音链路”仍然是核心能力，不应因桌面化而被弱化

无论宿主架构如何变化，以下主链路仍然是新系统必须继承的核心：

- 语音输入
- VAD / ASR
- 文本进入 LLM
- LLM 输出文本
- 一路进入 TTS 合成语音
- 一路转为角色动作 / 表情 / 状态反馈
- 最终驱动角色在直播中“说”和“动”

因此后续架构与模块设计中，不应让“桌面控制台”压过“主播主链路”本身。

控制台是为了让这条链路可控、可看、可干预，而不是替代它。

## 4. RAG 与商品消息的补充说明

## 4.1 本项目需要知识接入层，但当前不要求完整实现最终 RAG

目前阶段需要明确：

- 未来系统需要知识接入层
- 但当前阶段不要求确定最终向量库、Embedding 模型、检索框架和部署方案
- 当前阶段重点是先把边界与优先级设计清楚

---

## 4.2 必须区分两类知识来源

### 1. 长期知识（Long-term Knowledge）
例如：

- 角色设定
- 角色说话风格与限制
- 品牌信息
- 商品基础资料
- 常见问答
- 直播规则
- 固定运营知识

### 2. 临时高优先级直播上下文（Live Context / Product Feed）
例如：

- 当前主推商品
- 当前活动话术
- 当前优惠信息
- 当前库存提醒
- 临时禁说词
- 当前直播场次特殊要求
- 某一时段必须优先提及的卖点

---

## 4.3 设计要求

后续文档和实现中，必须明确以下原则：

- 长期知识可以检索
- 临时商品消息可以插入和替换
- 临时商品消息优先级高于长期知识
- 当前直播上下文应能被即时更新
- 不要把“商品消息”简单当作普通知识库文档
- LLM 模块不应独自承担知识层的全部职责
- 更合理的做法是为知识层 / 上下文层预留独立边界

---

## 5. 事件总线之外，还需要一个运行时控制器

当前项目不能只有事件流，而没有“谁负责系统当前运行状态”的模块。

因此建议在现有设计中明确承认一个独立角色：

**Runtime Controller / Orchestrator / Runtime Manager**

命名可后续再定，但职责必须清楚。

它至少应负责：

- 全局运行状态管理
- 串行处理规则
- 锁麦 / 解锁
- 急停 / 恢复
- 人工接管优先级
- 自动链路与人工链路之间的协调
- 决定当前是否允许继续自动播报、动作、响应事件

也就是说：

- 事件总线负责“发生了什么”
- 运行时控制器负责“当前允许系统怎么动”

如果没有这一层，状态逻辑会散落到多个模块中，后续会越来越乱。

---

## 6. 外部输入应被视为独立能力，而不只是“顺手加个事件”

未来外部输入可能来自：

- 直播平台弹幕
- 礼物 / 互动消息
- 商品切换消息
- 本地调试注入
- 平台 API
- 运营台消息
- 测试脚本

因此建议后续设计中明确保留一个独立边界，例如：

- `external-input`
- `platform-input`
- `live-input`

名称可调整，但其职责应独立存在。

这个模块至少应负责：

- 接收外部来源的事件
- 进行标准化
- 转换为内部统一事件
- 为日志、调试、回放留痕
- 与商品消息 / 平台事件解耦

不要把所有外部输入逻辑零散塞进控制台、LLM、角色模块中。

---

## 7. 高优先级口型数据不应与普通业务事件等同处理

虽然从抽象上说，嘴型驱动数据也可以被看作事件，

但在实现层面，应意识到：

- 嘴型数据通常是高频流
- 更新频率远高于普通业务事件
- 不适合完全按普通事件一样做全局广播和日志记录

因此建议在后续设计中明确：

- 高优先级、高频音频驱动数据可走专门通道或局部消费路径
- 不应让普通日志流、调试面板、事件面板被口型数据刷满
- 抽象上可以统一命名，但实现上应保留性能与可观测性的差异化处理空间

当前阶段不要求实现，但文档中应避免将其等同为普通控制事件。

## 8. Phase 文档生成策略补充

为了减少无效展开与频繁返工，后续阶段文档采用以下策略：

### 原则
- **每次最多只详细生成下一阶段文档**
- 不一次性细化所有 phase
- 高层路线图可以存在
- 但详细施工文档只为最近将要执行的阶段生成

### 原因
- 当前阶段很多约束仍在收口
- 过早细化后续阶段，容易把尚未确认的设定提前固化
- 每阶段结束后再生成下一阶段文档，更贴近真实项目推进

### 当前建议
- Phase 0 完成修订后，只生成 **Phase 1** 的详细文档
- 暂不生成 Phase 2 及之后的详细施工文档

## 9. Phase 0 修订目标

在阅读本文档后，Phase 0 需要做的不是推翻已有成果，而是**有针对性地修订**。

建议优先修订以下文档：

- `blueprints/phase0/architecture.md`
- `blueprints/phase0/module-design.md`
- `blueprints/phase0/data-flow.md`
- `blueprints/phase0/phase-roadmap.md`

修订重点包括：

### `architecture.md`
- 将“单进程架构”修正为“单桌面宿主应用 + 可插拔外部服务”
- 明确“单一状态真源，双窗口同步渲染”
- 明确 Windows-only 当前约束
- 明确桌面宿主的整合职责，而非一切逻辑都在宿主内实现

### `module-design.md`
- 补足 `runtime-controller` 或等价模块
- 补足 `knowledge` / `live-context` / `external-input` 中至少部分关键边界
- 防止知识、商品消息、编排逻辑都挤进 `llm` 模块
- 体现角色状态真源的设计原则

### `data-flow.md`
- 明确高频嘴型数据与普通业务事件的区别
- 明确外部商品消息与长期知识的不同注入路径
- 明确 runtime-controller 在链路中的控制作用
- 明确急停、恢复、人工接管时的流转规则

### `phase-roadmap.md`
- 保持总体 phase 结构
- 但只详细展开下一阶段
- 在下一阶段中增加必要的技术 spike / 验证任务
- 避免把未验证问题写成必然可行的正式实现路径

**`blueprints/phase1/foundation.md`**

- 基础事件总线与 runtime-controller 的最小可行形态

# Paimon Live — 架构设计

---

## 1. 总体架构

Paimon Live 是一个基于 Tauri 的桌面应用，采用**单桌面宿主 + 可插拔外部服务**的架构模型。

用户只需启动一个桌面程序，由该宿主统一承担整合、监控、控制、配置与启动职责。外部服务（ASR、TTS、LLM 等）由宿主统一管理和屏蔽复杂度，不暴露给非技术操作者。

```
┌─────────────────────────────────────────────────────┐
│                   Tauri 桌面宿主 (Rust)              │
│  窗口管理 · IPC 桥接 · 系统能力 · 外部服务管理 · 打包  │
├─────────────────────────────────────────────────────┤
│                 TypeScript 业务层                     │
│  运行时控制器 · 事件总线 · LLM 服务 · 音频管线         │
│  角色状态管理 · 知识/上下文层 · 外部输入 · 日志服务     │
├─────────────────────────────────────────────────────┤
│                    React UI 层                       │
│  控制面板 · 对话面板 · Live2D 渲染 · OBS 舞台窗口     │
└─────────────────────────────────────────────────────┘
         ↕ HTTP/HTTPS              ↕ HTTP
   ┌──────────┐              ┌──────────┐
   │ ASR 服务  │              │ TTS 服务  │
   │ (云端API) │              │(GPT-SoVITS)│
   └──────────┘              └──────────┘
         ↕ HTTP/HTTPS
   ┌──────────┐          ┌────────────────┐
   │ LLM 服务  │          │ 未来可能的       │
   │(OpenAI兼容)│          │ sidecar/helper │
   └──────────┘          └────────────────┘
```

### 核心设计原则

1. **单桌面宿主**：用户只启动一个桌面程序，宿主负责整合所有能力。但不强行承诺所有模块永远运行在同一系统进程里——允许外部服务、sidecar、helper process 存在，由宿主统一管理。
2. **TypeScript 优先**：核心业务逻辑（事件编排、LLM 对接、音频管线、角色状态）全部在 TypeScript 中实现。
3. **Rust 最小化**：Rust 仅承担 Tauri 宿主层职责（窗口、IPC、文件系统、打包），不承载业务逻辑。
4. **事件驱动 + 运行时控制**：模块间通过事件总线解耦通信，由运行时控制器统一管理系统运行状态（急停、锁麦、人工接管等）。
5. **单一状态真源**：角色状态只有一个权威数据源，所有窗口基于同一状态同步渲染。

## 2. 窗口设计

Paimon Live 需要至少两个窗口：

### 2.1 主控制台窗口

- 标准桌面窗口，不透明背景
- 承载控制面板、对话面板、角色预览、系统状态监控
- 操作员的主要交互界面

### 2.2 OBS 舞台窗口

- 透明背景窗口（Tauri 透明窗口 + 无装饰模式）
- 仅显示 Live2D 角色和可选字幕
- 供 OBS 窗口捕获使用
- 无 UI 控件干扰

### 2.3 单一状态真源，双窗口同步渲染

两个窗口并非共享同一个渲染器实例，而是：

- **角色状态只有一个权威真源**（`services/character` 管理）
- 表情、动作、嘴型、角色当前状态等数据来自同一个 authoritative state
- 主控界面预览与 OBS 舞台窗口各自有独立的 Live2D 渲染实例，但均基于同一状态源同步渲染
- 确保控制台看到的角色状态与 OBS 输出始终一致

## 3. 分层职责

### 3.1 Tauri 宿主层 (Rust)

| 职责 | 说明 |
|------|------|
| 窗口管理 | 创建/管理主窗口和 OBS 舞台窗口，支持透明背景 |
| IPC 桥接 | 提供 Tauri Command 供前端调用系统能力 |
| 文件系统 | 配置文件读写、日志文件输出 |
| 外部服务管理 | 未来可能管理 sidecar / helper process 的启停 |
| 系统托盘 | 可选的托盘图标与快捷操作 |
| 打包分发 | Windows 安装包生成 |

IPC 命令示例（后续 phase 实现）：
- 读写本地配置文件
- 打开/关闭 OBS 舞台窗口
- 获取系统音频设备列表

### 3.2 TypeScript 业务层

| 模块 | 职责 |
|------|------|
| runtime | 全局运行状态管理、串行处理规则、急停/恢复、人工接管、锁麦协调 |
| event-bus | 模块间事件发布/订阅，核心通信枢纽 |
| llm | LLM API 调用、对话管理、工具调用解析、流式响应 |
| audio | 麦克风采集、VAD、ASR 调用、TTS 调用、音频播放 |
| character | 角色人设加载、角色状态权威真源、表情/动作映射 |
| knowledge | 知识接入层——长期知识管理与临时高优先级上下文注入 |
| external-input | 外部事件标准化接入（弹幕、礼物、商品消息、调试注入等） |
| logger | 统一日志接口、日志分级、输出管理 |

业务层不直接操作 DOM，通过 React 状态机制驱动 UI 更新。

### 3.3 React UI 层

| 模块 | 职责 |
|------|------|
| live2d | PIXI + Live2D Cubism4 渲染，口型同步 |
| chat | 对话消息展示、文本输入 |
| control-panel | 系统状态监控、手动控制、配置调整 |
| stage | OBS 透明输出窗口的角色渲染 |

## 4. 与外部服务的通信

外部服务通信由 TypeScript 业务层发起。宿主应用屏蔽外部服务的部署细节，对操作者透明。

```
TypeScript 业务层
  ├── fetch() ──→ LLM API (OpenAI 兼容, HTTPS)
  ├── fetch() ──→ ASR API (SiliconFlow, HTTPS)
  └── fetch() ──→ TTS API (GPT-SoVITS, HTTP)
```

### 4.1 LLM 通信

- 协议：HTTPS，OpenAI Chat Completions 格式
- 流式：SSE（Server-Sent Events）接收流式回复
- 工具调用：function calling 格式，解析后通过事件总线分发

### 4.2 ASR 通信

- 协议：HTTPS POST
- 输入：音频 Blob（WAV/PCM）
- 输出：识别文本

### 4.3 TTS 通信

- 协议：HTTP POST（GPT-SoVITS 本地/远程服务）
- 输入：文本 + 角色语音配置
- 输出：WAV 音频数据

### 4.4 外部服务管理策略

- 当前阶段外部服务（ASR、TTS、LLM）均由用户自行部署或使用云端 API
- 未来宿主可通过 Tauri sidecar 能力管理本地服务的启停
- 桌面宿主负责检测外部服务连接状态并在控制面板中展示

### 4.5 未来扩展

- 弹幕/礼物：直播平台 WebSocket 或 HTTP 轮询（通过 external-input 模块接入）
- 商品消息：HTTP API 或 WebSocket 推送（通过 knowledge 模块注入）

## 5. 状态管理策略

### 5.1 状态分类

| 类别 | 示例 | 管理方式 |
|------|------|----------|
| 运行时状态 | 当前运行模式、是否急停、是否人工接管 | services/runtime 集中管理 |
| 角色状态 | 当前情绪、正在说话、活跃模型 | services/character 集中管理（权威真源） |
| 对话状态 | 消息历史、输入中、等待回复 | services/llm 管理 |
| 音频状态 | 录音中、VAD 激活、TTS 播放中、锁麦 | services/audio 管理 |
| 知识状态 | 长期知识索引、当前高优先级上下文 | services/knowledge 管理 |
| 系统状态 | 各服务连接状态、错误信息 | services/logger + 各服务自报 |
| UI 状态 | 面板展开/折叠、选中标签 | React 组件局部状态 |

### 5.2 状态共享

- 业务层各 service 维护自身状态，通过事件总线通知变化
- React 组件通过自定义 Hook 订阅 service 状态
- 避免在 React 组件中直接管理业务状态
- 运行时控制器拥有全局最高控制权，可覆盖其他模块行为（如急停时禁止新请求）

## 6. 安全与权限

### 6.1 Tauri 安全模型

- 使用 Tauri Capabilities 系统控制 IPC 命令权限
- CSP（Content Security Policy）配置允许连接外部 API
- 敏感配置（API Key）存储在本地配置文件中，不硬编码

### 6.2 紧急停止

- 全局快捷键触发紧急停止
- 由运行时控制器统一协调：中断 TTS 播放、暂停 LLM 请求、暂停音频采集
- 控制面板提供恢复操作
- 急停与恢复的状态转换由运行时控制器独占管理

---

## 7. 目标平台

- **仅 Windows**（Windows 10+）
- Tauri 打包为 `.msi` 或 `.exe` 安装包
- 不考虑 macOS / Linux 支持
- 如果未来要扩展平台，单独规划，不在当前阶段预支复杂度

# Paimon Live — 分阶段路线图

---

## 总览

项目按 phase 推进，每个 phase 有明确的边界和交付物。后一个 phase 严格依赖前一个 phase 的完成。

| Phase | 名称 | 核心目标 |
|-------|------|----------|
| 0 | Bootstrap | 项目骨架 + 审计 + 规划文档 |
| 1 | Foundation | Live2D 渲染 + 基础 UI + 事件总线 |
| 2 | Voice Pipeline | ASR + LLM + TTS 语音链路 |
| 3 | Control & Monitor | 控制面板 + 状态监控 + 紧急停止 |
| 4 | Live Integration | OBS 输出 + 弹幕/礼物接入 |
| 5 | Polish & Package | 配置管理 + 打包分发 + 体验打磨 |

---

## Phase 0 — Bootstrap（当前阶段）

**目标：** 搭建项目骨架，完成旧项目审计与架构规划。

**交付物：**
- Tauri + React + TypeScript 项目骨架
- 前端目录结构（features / services / hooks / types 等）
- VoiceL2D 审计报告
- 架构设计文档
- 模块设计文档
- 数据流与事件流设计文档
- 分阶段路线图

**不做的事：**
- 不接入真实服务
- 不迁移旧代码
- 不编写业务逻辑

---

## Phase 1 — Foundation

**目标：** 搭建可运行的基础框架——Live2D 角色能在窗口中渲染、事件总线能工作、基础 UI 结构就位。

**交付物：**
- 事件总线核心实现（发布/订阅、类型安全、订阅者管理）
- Live2D 渲染模块（Cubism Core 加载、模型加载、表情/动作播放）
- 角色状态管理基础实现
- 基础 UI 布局（主窗口分区：预览区 + 控制区 + 对话区）
- 日志服务基础实现
- 配置加载能力（读取本地配置文件）

**前置条件：**
- Phase 0 完成

**验收标准：**
- 应用启动后能看到 Live2D 角色在窗口中渲染
- 能通过代码或 UI 触发表情/动作切换
- 事件总线能正确发布和接收事件
- 日志能输出到控制台

**技术要点：**
- 引入 pixi.js + pixi-live2d-display
- 引入 Live2D Cubism Core
- 确定状态管理方案（Zustand / Jotai / 或自建轻量方案）
- 实现 Tauri IPC 命令：读取本地配置文件

---

## Phase 2 — Voice Pipeline

**目标：** 打通语音主链路，实现从用户说话到 AI 角色回复的完整闭环。

**交付物：**
- 音频采集模块（Web Audio API 麦克风输入）
- VAD 语音活动检测
- ASR 云端调用（SiliconFlow SenseVoice）
- LLM 对接（OpenAI 兼容 API，流式响应）
- TTS 调用（GPT-SoVITS HTTP API）
- 音频播放（播放队列 + 口型同步）
- 锁麦策略实现
- 对话面板基础 UI

**前置条件：**
- Phase 1 完成（事件总线、Live2D 渲染、角色状态管理可用）

**验收标准：**
- 用户对着麦克风说话，ASR 识别出文本
- LLM 根据角色人设生成回复
- TTS 合成语音并通过角色口型播放
- LLM 工具调用能触发角色表情变化
- 对话历史在面板中可见
- TTS 播放期间不会回声自触发

**技术要点：**
- 浏览器 MediaStream API 获取麦克风
- VAD 方案选型（@ricky0123/vad-web 或其他 WASM 方案）
- openai TS SDK 流式调用
- TTS 音频的 ArrayBuffer → AudioContext 播放
- Web Audio AnalyserNode 提取口型数据

---

## Phase 3 — Control & Monitor

**目标：** 提供完整的操作员控制台，支持监控系统状态、手动干预和紧急停止。

**交付物：**
- 控制面板完整 UI
  - 系统状态仪表盘（各服务连接/延迟/错误）
  - 手动触发表情/动作面板
  - 角色/模型切换
  - 音频设备选择
  - 配置编辑面板
- 紧急停止与恢复功能
- 人工接管模式（手动输入回复文本）
- 日志查看面板
- 全局快捷键支持

**前置条件：**
- Phase 2 完成（语音链路可用）

**验收标准：**
- 控制面板实时显示各服务状态
- 操作员可手动触发表情、动作
- 紧急停止能立即中断所有活动
- 人工接管模式下操作员可直接输入回复
- 全局快捷键能触发紧急停止

**技术要点：**
- Tauri 全局快捷键 API
- 服务健康检测机制
- 配置文件读写（Tauri IPC）

---

## Phase 4 — Live Integration

**目标：** 实现直播场景集成——透明 OBS 输出窗口、弹幕/礼物接入、商品消息注入。

**交付物：**
- OBS 舞台窗口（透明背景、无装饰、角色渲染同源）
- 可选字幕叠加
- 直播平台弹幕/礼物事件接入（至少支持一个主流平台）
- 商品消息接入（区分长期知识和临时高优先级消息）
- 外部事件接入适配器框架

**前置条件：**
- Phase 3 完成（控制面板可用）

**验收标准：**
- OBS 能捕获透明背景的角色输出窗口
- 舞台窗口角色状态与主窗口一致
- 弹幕消息能作为用户输入触发 AI 回复
- 商品消息能注入 LLM 上下文，且临时消息优先级高于长期知识

**技术要点：**
- Tauri 多窗口管理（透明窗口 + decorations: false）
- 直播平台 API/WebSocket 协议对接
- 知识注入的优先级排序机制
- 需要在 `docs/research/` 中调研目标直播平台接口规范

---

## Phase 5 — Polish & Package

**目标：** 体验打磨、稳定性提升、打包为可分发的 Windows 安装包。

**交付物：**
- 首次启动引导流程（配置 API Key、选择音频设备等）
- 错误恢复与自动重连
- 性能优化（渲染帧率、内存占用）
- UI 美化与交互细节打磨
- Windows 安装包（.msi / .exe）
- 用户文档

**前置条件：**
- Phase 4 完成

**验收标准：**
- 非技术人员能按引导完成首次配置
- 运行稳定，长时间不崩溃
- 安装包可在干净 Windows 系统上正常安装运行

**技术要点：**
- Tauri Bundle 配置
- 签名与安装包定制
- 性能 profiling 与优化

## 风险与待定事项

| 事项 | 风险等级 | 说明 |
|------|----------|------|
| VAD 方案选型 | 中 | 需在 Phase 2 前确定，可能需要调研多个方案 |
| 本地 ASR | 低 | 暂不纳入路线图，作为未来可选功能 |
| 直播平台接口 | 中 | 各平台接口差异大，需在 Phase 4 前调研 |
| Live2D 商业模型授权 | 中 | 需确认模型使用授权范围 |
| GPT-SoVITS 部署 | 低 | 作为外部服务，不在 Paimon Live 进程边界内 |
| 多角色并发 | 低 | 当前设计为单角色，多角色场景暂不考虑 |
